The Dopamine Machine

Every tap of "Open Pack" sets off a chain reaction across three computers — and it all finishes before you finish blinking. Here is how.

What is Dopamaxxing?

Dopamaxxing is a Pokémon card gacha game: spend virtual coins to open packs of cards, collect rare Pokémon, and show off your collection. It is the excitement of pulling a holographic Charizard, recreated entirely in software.

The whole app is built on three pillars — opening, selling, and battling. Everything else supports these three loops.

🎴
Open Packs

Spend coins, get random Pokémon cards. Pull odds vary by rarity — common cards fall out often, legendary ones almost never.

💰
Sell & Trade

Every card has a buyback value based on condition. Sell duplicates to earn more coins and keep the loop going.

⚔️
Battle Trainers

Use your card collection to fight AI trainers in turn-based combat. Your card stats determine the outcome.

💡
Key insight

The whole app is one big request-response loop: the browser asks, the server decides, the database remembers.

The Moment of the Click

Think of the app like a vending machine. When you tap "Open Pack," you trigger a chain reaction across multiple computers. Here is every step, in order.

1
You click "Open Pack" in your browser

Your finger taps a button rendered by React. Nothing has happened on the server yet.

2
Your browser sends a request to the server

The browser fires a POST request to /api/open-pack over HTTPS.

3
Server checks: Are you logged in? Got enough coins? Bag not full?

The authentication check happens first. If any condition fails, the server sends back an error and stops — no cards are given.

4
Server fetches cards from the database + stats from PokeAPI simultaneously

Using Promise.all(), three database queries and an external API call fire at the exact same time.

5
Server rolls the random card selection, calculates stats and worth

The RNG runs on the server — never the browser — so players cannot cheat by manipulating the odds.

6
Server deducts coins, saves cards to database

Both writes happen inside a single database transaction — ensuring you can never lose coins without getting cards.

7
Server sends cards back — your browser plays the reveal animation

The server returns a JSON response with the card data; the browser then runs the animation to reveal your new cards.

Code ↔ English Translation

Here is what the server actually does when it receives your pack request. This is real code from app/api/open-pack/route.ts:

CODE 
const [{ data: profile }, allCards, bagCountRes] =
  await Promise.all([

    supabase
      .from('profiles')
      .select('pity_counter, coins, xp,
        level, bag_capacity,
        daily_packs_today,
        packs_opened')
      .eq('id', user.id)
      .single(),

    packDef?.theme_pokedex_ids
      ? getCardsForPokedexIds(
          supabase, setId,
          packDef.theme_pokedex_ids)
      : getCardsForSet(
          supabase, setId),

    supabase
      .from('user_cards')
      .select('id', {
        count: 'exact',
        head: true
      })
      .eq('user_id', user.id),
  ])

const bagCapacity =
  profile?.bag_capacity ?? 50

const bagCount =
  bagCountRes.count ?? 0

if (bagCount >= bagCapacity) {
  return NextResponse.json(
    { error: 'bag_full',
      bagCount, bagCapacity },
    { status: 409 },
  )
}

Do three database lookups at the exact same time — parallel is faster than one-by-one.

Get every card available in this pack set.
Count how many cards are already in this user's bag.
(all three happen simultaneously — done in one database round-trip)
How many cards can this user hold? Default is 50 if not set.
Count how many cards they currently have.
If the bag is already full — stop right here.
Send back an error with code 409 (the universal code for “conflict — no room”).

Why does this matter?

Understanding which computer does each job means you can pinpoint bugs instantly. Coin balance wrong? Look at the API route. Animation glitchy? Look at the React component. They are separate concerns on separate computers.
