The app has distinct "actors." Each one has exactly one job. When something breaks, knowing who owns what means you fix it in 5 minutes instead of 5 hours.

The Three Layers

Every web app, no matter how complex, is built in layers. Dopamaxxing has three. Knowing which layer owns which job is the key to steering AI effectively.

Think of it like a film crew: you see the movie (the UI), but behind the scenes there is a director (the API routes), a props department (the database), and a cashier (Stripe). Each has one job and never does someone else's.

The Browser Actors

These are the four main React components that make up the visible app. Each one manages a specific slice of the user experience.

🎴
PackOpening.tsx

Animates the card reveal — the shake, the tear, the flip. Manages the entire pack-opening experience and calls /api/open-pack to get the actual cards from the server.

🎒
Bag.tsx

The card inventory. Shows all your cards and lets you filter by rarity, sort, sell, and feed cards. Bulk operations go through /api/batch-action.

⚔️
NBattleScreen.tsx

The full battle arena. Orchestrates all phases: pre-dialogue → team reveal → turn-based combat → win/loss screens.

🛍️
PackSelector.tsx

The pack shop. Shows all available packs with prices and pull odds. When you choose one, it hands the selection to PackOpening to begin the experience.

💡
Important boundary

These are all client components — they run in your browser, not on the server. They can play animations and respond to clicks, but they cannot directly access the database — they have to ask the server first.

The director rule

Notice that PackOpening never talks directly to Supabase or PokeAPI. The API route is always in the middle, coordinating everything. This is a core architectural rule: browsers talk to servers, servers talk to databases — never skip a layer.

Code ↔ English: What Auth Actually Means

Every single API route starts the same way — it checks who you are before doing anything. Here is what that looks like in real code from app/api/open-pack/route.ts.

CODE 
export async function POST(request: NextRequest) {
    try {
        const supabase = await createClient()
        const {
            data: { user },
        } = await supabase.auth.getUser()

        if (!user)
            return NextResponse.json(
                { error: 'Unauthorized' },
                { status: 401 }
            )
PLAIN ENGLISH
This function wakes up every time someone sends a POST request to /api/open-pack.
Connect to Supabase using the login cookies from the request.
Ask Supabase: who sent this? Are they logged in?
Get back the user object — their ID, email, and auth status.
If nobody is logged in (no valid cookie) — stop right here.
Send back an error: "Unauthorized." Code 401 is the universal internet code for this — every browser in the world understands it.
🚪
Authentication middleware

This pattern — checking identity before doing anything — is called authentication middleware. Every serious app does this on every protected endpoint. It is the server equivalent of a bouncer checking IDs — no valid ID, no entry.

💡
Why this matters for directing AI

When you ask an AI to add a new feature to an API route, you need to tell it: "put the new logic after the auth check." If you forget, the AI might generate code that runs before the user is verified — a serious security hole that is hard to spot.
